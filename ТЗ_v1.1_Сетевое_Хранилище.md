## 1. Обзор проекта

**HomeCloud** — самохостируемое сетевое хранилище, разворачиваемое на домашнем ПК через Docker. Предоставляет веб-интерфейс для управления файлами, фотографиями и зашифрованными документами нескольких пользователей с разными уровнями доступа.

### Ключевые принципы разработки

| Принцип | Применение |
|---|---|
| **Clean Architecture** | Разделение на слои: Domain → Application → Infrastructure → Presentation |
| **SOLID** | Каждый класс — одна ответственность, открыт к расширению |
| **DRY** | Разделы «Файлы» и «Приватное» реализуются через единый `FileManager` с обёрткой шифрования |
| **KISS** | Без избыточных абстракций; интерфейсы не обязательны |
| **Без тестов** | Код организуется так, чтобы тесты можно было добавить позже |

## 2. Ролевая модель

### 2.1 Роли пользователей

| Роль | Код | Описание |
|---|---|---|
| Незнакомец | `STRANGER` | Самостоятельная регистрация; ограниченный объём |
| Член семьи | `FAMILY` | Назначается администратором; расширенные права |
| Администратор | `ADMIN` | Полный доступ + управление пользователями |

### 2.2 Матрица прав доступа

| Возможность | STRANGER | FAMILY | ADMIN |
|---|:---:|:---:|:---:|
| Авторизация / регистрация | ✅ | ✅ | ✅ |
| Собственные фото | ✅ | ✅ | ✅ |
| Собственные файлы | ✅ | ✅ | ✅ |
| Приватный раздел | ✅ | ✅ | ✅ |
| Папка «Общее» | ❌ | ✅ | ✅ |
| Загрузка без шифрования | ✅ | ✅ | ✅ |
| Загрузка с шифрованием (приватное) | ✅ | ✅ | ✅ |
| Админ-панель | ❌ | ❌ | ✅ |
| Изменение ролей | ❌ | ❌ | ✅ |
| Настройка лимитов пользователей | ❌ | ❌ | ✅ |

### 2.3 Лимиты хранилища

```
STRANGER:
  - Общий объём: 100 МБ (жёстко, без возможности превышения)
  - Приватный раздел: входит в общий лимит; выделяемый подлимит задаётся в админке

FAMILY:
  - Общий объём: свободное место на диске (динамически)
  - Приватный раздел: N ГБ (задаётся в админке отдельно для каждого пользователя)

ADMIN:
  - Те же правила, что и FAMILY
```

- Создание папки:
  - Кнопка «Новая папка»
  - Диалоговое окно с полем ввода названия и кнопками «Создать» / «Отмена»
  - Валидация: недопустимые символы в имени папки
- Сортировка: по имени, дате, размеру
- Отображение размера файлов и дат изменения

### 3.6 Страница «Приватное»

**URL:** `/private`

#### Вход в раздел

- При переходе на страницу — **модальное окно** с полем ввода кодового слова (пароля шифрования)
- Кодовое слово хранится только в **памяти сессии** (не в localStorage, не в cookie, не в БД)
- При закрытии вкладки / истечении сессии — кодовое слово уничтожается
- Без ввода кодового слова — доступ к разделу закрыт

#### Функционал

- Идентичен разделу «Файлы» (используется тот же компонент `FileManager` с `mode: 'encrypted'`)
- Дополнительно в шапке раздела: `[занято в приватном] из [лимит приватного]`
- Все имена файлов и папок **шифруются** (хранятся зашифрованными на диске)
- Содержимое файлов **шифруется** перед записью на диск
- При отображении — расшифровывается на лету (потоково для больших файлов)

#### Поведение ключа шифрования

```
1. Пользователь вводит кодовое слово → POST /api/private/unlock (ключ в теле запроса по HTTPS)
2. Сервер валидирует ключ (попытка расшифровать «маркерный» файл)
3. Если ключ верный — производный ключ сохраняется в серверной сессии (in-memory/Redis, TTL = PRIVATE_SESSION_TTL_HOURS)
4. Ключ НЕ пишется в БД, НЕ пишется в файл, НЕ логируется
5. При истечении TTL приватной сессии — ключ уничтожается; основная сессия остаётся активной
6. Все последующие операции берут ключ из сессии
7. TTL сбрасывается (sliding window) при каждом действии в приватном разделе
```

> **Для реализации:** Приватная сессия имеет **отдельный, более короткий TTL** от основной JWT-сессии. Переменная окружения: `PRIVATE_SESSION_TTL_HOURS` (по умолчанию 4). Хранилище сессии должно поддерживать независимое управление TTL для приватного ключа и основного токена. При истечении приватного TTL клиент получает код `PRIVATE_SESSION_EXPIRED` (HTTP 401) и показывает диалог повторного ввода кодового слова — без полного выхода из системы.

## 4. Файловая система и хранилище

### 4.1 Структура папок на диске

```
/storage/                        ← корень (маунтится в Docker)
├── disk1/                       ← первый диск (volume)
│   ├── _meta/                   ← служебные файлы (маркеры, индексы)
│   ├── shared/                  ← общая папка (FAMILY + ADMIN)
│   │   ├── photos/
│   │   └── files/
│   └── users/
│       ├── {user_id}/
│       │   ├── photos/          ← оригиналы + preview
│       │   │   ├── originals/
│       │   │   └── previews/    ← thumbnail (генерируется при загрузке)
│       │   ├── files/           ← пользовательские файлы
│       │   │   └── ...          ← произвольная вложенность папок
│       │   └── private/         ← зашифрованные файлы
│       │       ├── .marker      ← маркерный файл для валидации ключа
│       │       └── ...          ← зашифрованные файлы и папки
│       └── {user_id}/
│           └── ...
└── disk2/                       ← второй диск (подключается позже)
    ├── _meta/
    ├── shared/
    └── users/
```

### 4.2 Хранение метаданных файлов

Метаданные файлов хранятся в **базе данных** (не только в файловой системе):

```
FileRecord {
  id: UUID
  user_id: UUID
  disk_id: string          ← на каком диске лежит файл
  relative_path: string    ← путь относительно корня диска
  original_name: string    ← оригинальное имя (для зашифрованных — зашифровано)
  size_bytes: long
  mime_type: string
  is_encrypted: bool
  section: enum(PHOTOS, FILES, PRIVATE, SHARED)
  status: enum(PENDING, COMMITTED, ARCHIVED, DELETED)  ← жизненный цикл записи
  checksum_sha256: string | null   ← контрольная сумма для проверки целостности
  created_at: datetime
  last_accessed_at: datetime
  is_archived: bool
  archive_path: string?
}
```

> **Примечание по статусу:** `PENDING` — файл в процессе загрузки; `COMMITTED` — файл успешно записан и верифицирован; `DELETED` — помечен на удаление (soft delete). Файлы со статусом `PENDING` старше 1 часа считаются «зависшими» и подлежат очистке фоновой задачей. Это необходимо для реализации атомарной записи в будущих версиях.

## 5. Шифрование и безопасность

### 5.1 Алгоритм шифрования

- **Алгоритм:** AES-256-GCM (аутентифицированное шифрование)
- **Ключ:** производится из кодового слова через **PBKDF2** (SHA-256, 100 000 итераций, соль = user_id)
- **IV/Nonce:** генерируется случайно для каждого файла, хранится в первых байтах зашифрованного файла

### 5.2 Шифрование имён файлов и папок

- Имя файла шифруется и кодируется в **Base64URL**
- В БД хранится зашифрованное имя
- Расшифровка происходит при каждом запросе (на сервере, ключ берётся из сессии)

### 5.3 Маркерный файл

При первом создании приватного раздела (первый `unlock`):

```
Создаётся файл: /storage/disk_N/users/{user_id}/private/.marker
Содержимое: encrypt("HOMECLOUD_MARKER_V1", key)
```

При каждом последующем входе: расшифровать `.marker` и сравнить с ожидаемой строкой.

### 5.4 Передача данных

- Весь трафик — только через **HTTPS** (самоподписанный сертификат или Let's Encrypt)
- Кодовое слово передаётся только в теле POST-запроса (не в URL, не в заголовках)
- JWT-токены — в **httpOnly, Secure, SameSite=Strict** cookie

## 6. Архитектура приложения

### 6.1 Общая схема слоёв (Clean Architecture)

```
┌─────────────────────────────────────────────────────┐
│                  Presentation Layer                  │
│         (HTTP Handlers / REST Controllers)           │
│              + Frontend (React / Jinja)              │
└──────────────────────┬──────────────────────────────┘
                       │ вызывает
┌──────────────────────▼──────────────────────────────┐
│                 Application Layer                    │
│           (Use Cases / Service Classes)              │
│  AuthService, FileService, PhotoService,             │
│  PrivateService, AdminService, ArchiveService        │
└──────────────────────┬──────────────────────────────┘
                       │ использует
┌──────────────────────▼──────────────────────────────┐
│                  Domain Layer                        │
│        (Entities, Value Objects, Domain Rules)       │
│  User, Role, FileRecord, StorageQuota, DiskVolume   │
└──────────────────────┬──────────────────────────────┘
                       │ реализует
┌──────────────────────▼──────────────────────────────┐
│               Infrastructure Layer                   │
│  DiskRouter, PlainStorageAdapter,                    │
│  EncryptedStorageAdapter, ThumbnailService,          │
│  ArchiveManager, DB (SQLAlchemy / Prisma),           │
│  SessionStore, GoogleOAuthClient                     │
└─────────────────────────────────────────────────────┘
```

### 6.2 Описание слоёв

#### Domain Layer

Не зависит ни от чего. Содержит:

- `User` — сущность пользователя (id, email, role, quota settings)
- `FileRecord` — сущность файла (метаданные)
- `DiskVolume` — сущность диска (id, mount_path, priority)
- `StorageQuota` — расчёт и проверка лимитов
- `Role` — перечисление ролей и правила доступа

#### Application Layer

Оркестрирует операции, не знает о HTTP и файловой системе напрямую:

- `AuthService` — регистрация, вход, Google OAuth, JWT
- `FileService` — CRUD файлов и папок (делегирует в `StorageAdapter`)
- `PhotoService` — загрузка фото, генерация превью, пагинация
- `PrivateService` — разблокировка, управление ключом в сессии
- `AdminService` — управление пользователями и квотами
- `ArchiveService` — архивирование и разархивирование

#### Infrastructure Layer

- `DiskRouter` — выбор диска для записи (см. [раздел 9](#9-расширяемость-по-дискам)); проверяет физическую доступность диска перед возвратом; кэширует свободное место
- `PlainStorageAdapter` — работа с файловой системой без шифрования; все пути нормализуются и проверяются на path traversal перед использованием
- `EncryptedStorageAdapter` — обёртка над `PlainStorageAdapter`, добавляет шифрование
- `ThumbnailService` — генерация превью изображений (Pillow / Sharp); **вызывается только из фонового контекста**, не из синхронного обработчика запроса
- `ArchiveManager` — сжатие через zstd; принимает параметр `compress_mode: 'pre_encrypt' | 'post_encrypt'`
- `SessionStore` — хранение кодового слова в серверной сессии (Redis или in-memory); поддерживает раздельный TTL для приватного ключа
- `OAuthClient` — интеграция с Google OAuth 2.0
- `Database` — ORM-модели и репозитории

**Репозитории:**
- `UserRepository`
- `FileRepository`
- `QuotaRepository` — работа с денормализованной таблицей квот; поддерживает атомарный инкремент/декремент
- `UploadSessionRepository` — хранение состояния загрузки (зарезервировано для chunked upload)

#### Presentation Layer

- REST API (FastAPI / Express) — эндпоинты для всех операций
- Frontend — React SPA или Server-Side Jinja-шаблоны

| Thumbnail | **Pillow** | Стандарт для Python |
| Архив | **zstandard** | Высокая степень сжатия без потерь |
| Контейнер | **Docker + docker-compose** | Изоляция и портируемость |

### 6.4 Паттерн переиспользования `FileManager`

Ключевое решение для разделов «Файлы» и «Приватное» — **паттерн Decorator (обёртка)**:

```
StorageAdapter (базовый класс)
├── PlainStorageAdapter       ← работает с ФС напрямую
└── EncryptedStorageAdapter   ← принимает PlainStorageAdapter + ключ
    ├── write(path, data)     → encrypt(data) → plain.write(encrypted_path, encrypted_data)
    ├── read(path)            → plain.read(encrypted_path) → decrypt(data)
    ├── list(path)            → plain.list() → decrypt каждое имя
    ├── mkdir(path)           → plain.mkdir(encrypt(path))
    └── delete(path)          → plain.delete(encrypted_path)
```

**Использование в Application Layer:**

```python
# Раздел "Файлы"
file_service = FileService(
    adapter=PlainStorageAdapter(base_path=user_files_dir)
)

# Раздел "Приватное"
private_service = FileService(
    adapter=EncryptedStorageAdapter(
        inner=PlainStorageAdapter(base_path=user_private_dir),
        key=session.private_key  # берётся из серверной сессии
    )
)
```

`FileService` не знает, с каким адаптером работает — он универсален. Добавление нового типа хранилища = создание нового адаптера без изменения сервиса.

**Frontend:**

```tsx
// Один компонент FileManager, два использования
<FileManager mode="plain"     apiPrefix="/api/files"   />
<FileManager mode="encrypted" apiPrefix="/api/private" />
```

## 9. Расширяемость по дискам

### 9.1 Концепция DiskRouter

`DiskRouter` — компонент Infrastructure Layer, ответственный за выбор диска при записи нового файла.

```python
class DiskVolume:
    id: str               # "disk1", "disk2"
    mount_path: Path      # /storage/disk1
    priority: int         # чем меньше, тем приоритетнее
    is_active: bool

class DiskRouter:
    def get_write_disk(self) -> DiskVolume:
        """
        Возвращает диск с наибольшим свободным местом
        среди активных и доступных дисков.
        Стратегия: Most Free Space First.
        Перед возвратом проверяет физическую доступность диска (os.statvfs).
        При недоступности всех дисков — выбрасывает StorageUnavailableError.
        """

    def get_all_disks(self) -> list[DiskVolume]:
        """Список всех активных дисков."""

    def get_disk_by_id(self, disk_id: str) -> DiskVolume:
        """Для чтения файла с конкретного диска."""

    def health_check(self) -> dict[str, str]:
        """
        Возвращает статус каждого диска: HEALTHY | LOW_SPACE | UNAVAILABLE.
        Используется в /api/admin/storage/health и фоновым мониторингом.
        Результат кэшируется на DISK_HEALTH_CACHE_TTL секунд.
        """

    def get_free_space(self, disk_id: str) -> int:
        """
        Возвращает свободное место на диске в байтах.
        Результат кэшируется на DISK_SPACE_CACHE_TTL секунд (по умолчанию 30).
        """
```

### 9.2 Добавление нового диска (процедура)

```
1. Физически подключить новый диск к ПК
2. Смонтировать папку нового диска в docker-compose.yml:
     - /path/to/disk2:/storage/disk2
3. Добавить disk2 в переменную окружения STORAGE_DISKS=disk1,disk2
4. Перезапустить контейнер: docker-compose up -d
5. DiskRouter автоматически обнаружит новый диск и начнёт писать на него
```

Перемещение файлов со старого диска **не требуется** — чтение всегда идёт по `disk_id`, хранящемуся в метаданных (`FileRecord.disk_id`). Новые файлы пишутся на диск с наибольшим свободным местом.

### 9.3 Стратегии выбора диска (расширяемо)

```python
class DiskSelectionStrategy:  # base class
    def select(self, disks: list[DiskVolume]) -> DiskVolume: ...

class MostFreeSpaceStrategy(DiskSelectionStrategy): ...   # по умолчанию
class RoundRobinStrategy(DiskSelectionStrategy): ...      # равномерное распределение
class PriorityStrategy(DiskSelectionStrategy): ...        # по полю priority
```

Стратегия задаётся через конфигурацию (`DISK_STRATEGY=most_free_space`).

## 10. Архивирование

### 10.1 Концепция

Файлы и фотографии, к которым не обращались более N дней (настраивается через `ARCHIVE_DAYS_THRESHOLD`), переносятся в архив.

### 10.2 Процесс архивирования

```
1. Фоновая задача (Celery / APScheduler) раз в сутки:
   - Запрос к БД: FileRecord.last_accessed_at < now() - threshold AND status = COMMITTED
   - Для каждого файла:
     a. Сжать файл: zstandard (уровень 22 — максимальное сжатие)
     b. Сохранить архив: original_path + ".zst"
     c. Удалить оригинал
     d. Обновить FileRecord: is_archived=True, status=ARCHIVED, archive_path="..."

2. При запросе архивного файла:
   - Прозрачная распаковка на лету (потоково)
   - Обновление last_accessed_at
   - Опционально: восстановление оригинала (если часто запрашивается)
```

### 10.3 Зашифрованные файлы в архиве

```
Режим compress_mode определяет порядок операций:

  pre_encrypt  (для обычных файлов — максимальная степень сжатия):
    Сжать → Зашифровать → записать .zst.enc

  post_encrypt (для приватных файлов — архивируется уже зашифрованный контент):
    Зашифровать (уже сделано при сохранении) → Сжать → записать .enc.zst

Порядок при чтении (post_encrypt):
  1. Распаковать → 2. Расшифровать (ключ из сессии) → 3. Отдать клиенту
```

> **Важно:** В режиме `post_encrypt` фоновый worker не требует ключа шифрования — он работает с уже зашифрованным файлом как с обычными байтами. Это позволяет автоматически архивировать приватные файлы без доступа к ключу пользователя. Коэффициент сжатия в этом режиме ниже (зашифрованные данные имеют высокую энтропию), что является ожидаемым компромиссом.

> **Параметр `compress_mode`** должен быть явно указан в `ArchiveManager` и задаваться на уровне вызова из `ArchiveService`. Это точка расширения для будущих изменений стратегии архивирования.

## 13. Нефункциональные требования

### 13.1 Производительность

- Загрузка страницы фото (первый батч) — не более 2 секунд
- Генерация превью — асинхронно (не блокирует ответ на загрузку)
- Большие файлы (>100 МБ) — потоковая передача (streaming upload/download)

### 13.2 Безопасность

- Все эндпоинты (кроме `/auth`) — защищены JWT-авторизацией
- Проверка прав доступа на уровне Application Layer (не только в роутерах)
- **Защита от path traversal:** все пути файловой системы нормализуются через `Path.resolve()`, затем проверяется, что результирующий путь начинается с разрешённого корня пользователя. Запросы с `..` в любом сегменте отклоняются с HTTP 400 до нормализации
- Rate limiting на эндпоинты авторизации
- **Rate limiting на `/api/private/unlock`:** максимум 5 неудачных попыток за 15 минут с одного IP; реализуется через счётчик в Redis
- Входные данные валидируются через Pydantic (backend) и Zod (frontend)
- **Коды ошибок API:** все ошибки возвращают структурированный JSON `{error_code: string, message: string}`. Клиент должен показывать пользователю осмысленное сообщение, а не технический текст ошибки. Список обязательных кодов: `QUOTA_EXCEEDED`, `UNSUPPORTED_FORMAT`, `PRIVATE_SESSION_EXPIRED`, `DISK_UNAVAILABLE`, `PATH_TRAVERSAL_DETECTED`, `FILE_NOT_FOUND`, `ACCESS_DENIED`

### 13.3 Расширяемость

- Добавление нового раздела (например, «Музыка») = новый роутер + новая страница + тот же `FileManager`
- Добавление нового диска — без остановки и изменения кода (только конфигурация)
- Замена алгоритма шифрования — только в `EncryptedStorageAdapter`
- Замена стратегии выбора диска — только в `DiskRouter`

### 13.4 Адаптивность UI

- Mobile-first подход
- Breakpoints: 640px (sm), 1024px (lg)
- Тестирование на: iPhone Safari, Android Chrome, Desktop Chrome/Firefox

### 13.5 Логирование

- Все операции с файлами логируются (без содержимого файлов и ключей шифрования)
- Логи доступны через `docker logs` и опционально в файл
- **Обязательные поля лога:** `timestamp`, `user_id`, `action`, `section`, `file_id` (если применимо), `disk_id`, `result (success | error)`, `error_code`
- Логи ошибок не должны содержать пути файлов приватного раздела в открытом виде

## 14. Точки расширения (зарезервировано)

Этот раздел описывает компоненты, которые **в v1 реализуются в минимальном виде**, но спроектированы так, чтобы их можно было расширить без рефакторинга смежного кода.

| Компонент | Текущее поведение (v1) | Подготовлено для расширения |
|---|---|---|
| `UploadSessionRepository` | Не используется; файл существует | Chunked / resumable upload (TUS) |
| `FileRecord.status` | `COMMITTED` сразу при загрузке | Атомарная запись через `PENDING → COMMITTED` |
| `FileRecord.checksum_sha256` | Вычисляется, сохраняется | Периодический scrubbing целостности |
| `DiskRouter.health_check()` | Метод есть, но не вызывается | Мониторинг дисков, уведомления о LOW_SPACE |
| `ArchiveManager.compress_mode` | Параметр есть; `pre_encrypt` для всех | `post_encrypt` для приватных файлов |
| `ThumbnailService` | Вызывается синхронно | Перенос в очередь задач (async worker) |
| `/api/admin/storage/health` | Endpoint есть, возвращает заглушку | Реальный health check дисков |
| `/api/files/search` | Endpoint есть, возвращает 501 | Полнотекстовый поиск по именам |
| `QuotaRepository` | Денормализованная таблица с атомарным incr | Ежесуточная сверка с реальным `SUM` |
| `ErrorMessage` (frontend) | Отображает `error_code` и message | Детальные инструкции по устранению |
| `SessionStore` | Единый TTL | Раздельный TTL для приватного ключа |

> **Правило:** Любой из пунктов этой таблицы может быть реализован как отдельная задача без изменения контракта смежных компонентов. Это является критерием корректной реализации v1.
